# AGENTS.md — AI Agent 工作规范

> 本文档定义本项目所有 AI Agent（无论由哪位团队成员使用）必须遵循的行为规范。
> **这不是项目介绍**，而是 AI 的工作纪律。

---

## 1. 项目身份

这是一个**大一学生展示型项目**，5 人协作，5 天开发周期，纯前端 Web 应用。追求的是：

- **可演示**——评审/展示时能完整跑通
- **可维护**——5 个人改代码不会互相踩
- **可协作**——新人能立刻上手

不是生产级系统。不要引入生产系统的复杂度。

---

## 2. 核心原则

### 2.1 第一原则

**清晰、稳定、可演示 > 复杂架构 > "最佳实践"。**

当一个设计选择面临权衡时，永远选最清晰的那个，而非最"优雅"的那个。

### 2.2 不要做的事

- 不要引入不必要的抽象层、设计模式或过度工程化
- 不要为了"代码更优雅"而大规模重构已有代码
- 不要实现 PRD 未定义的功能（除非用户明确要求）
- 不要强推（force push）、重写 Git 历史或删除重要文件
- 不要基于假设修改代码——先阅读，再动手

---

## 3. 文档层级与读取规则

### 3.1 优先级

```
PRD_OpenCheck.md  >  WORK_DIVISION.md  >  其他共享文档
  （产品需求）          （分工+计划）          （/docs/*）
```

- 产品需求以 **PRD_OpenCheck.md** 为准
- 分工与开发计划以 **WORK_DIVISION.md** 为准
- **AGENTS.md** 仅规定 AI 工作规范，不涉及具体项目需求
- 文档之间如有冲突，**不要自行判定**——提醒用户确认

### 3.2 按需分层读取

| 层级 | 包含什么 | 何时读取 |
|------|---------|---------|
| **第一层（默认必读）** | AGENTS.md、PRD_OpenCheck.md、WORK_DIVISION.md | 每个新会话开始时 |
| **第二层（按需）** | `/docs/` 下的共享设计文档、CONTRACTS.md | 当前任务涉及对应主题时 |
| **第三层（默认不读）** | `/docs/notes/R1/` 至 `/docs/notes/R5/` | 仅在当前任务明确涉及对应角色时 |

**先读需要的内容，信息不足时再继续读取。不要一次性把全部文档读完。**

### 3.3 长期文档 vs 临时文档

| | 长期稳定文档 | 临时/个人文档 |
|--|------------|-------------|
| **位置** | `/docs/` | `/docs/notes/Rx/` |
| **内容** | 架构、接口、共享类型、公共规范 | 开发计划、思路草稿、实验记录、个人 TODO |
| **维护者** | 相关角色所有者 | 个人自由管理 |
| **生命周期** | 随项目长期维护 | 开发完成后可丢弃 |

## 4. 开发前检查

### 4.1 多人协作基线

开始任何开发任务前，先做两步检查：

1. **查看最近的 commit 记录**——确认你即将修改的文件在远端是否有更新，避免基于过时状态工作
2. **确认公共文档、代码、Commit History 三者一致**——发现不一致时优先修复文档

### 4.2 任务分解

开发顺序以 **WORK_DIVISION.md 的 5 日计划**为骨架。

每次实现任务前，**先说计划和影响范围**，用户确认后再动手。不要默默写一堆代码。

---

## 5. 代码实施规范

### 5.1 契约稳定性

以下内容**冻结后未经用户确认不得修改**：

- 共享类型定义（`src/types/`）
- 公共函数/组件签名
- ROUTE 常量
- localStorage 存储 key 和数据结构

具体契约定义见 `/docs/CONTRACTS.md`（如有）。如果一个函数/类型被其他角色引用，改动前必须在团队群通知。

### 5.2 修改已有代码

修改已有代码前，必须先阅读相关实现，理解现有逻辑。不允许"这个函数看起来不对，我直接重写一个"。

### 5.3 不确定时

需求或文档不明确 → **向用户确认**，不要自行猜测设计意图。

技术方案有多个选项，且选哪个会影响其他人 → **向用户确认**。

### 5.4 风格一致

保持与项目已有代码一致：

- 命名风格（变量、函数、文件）
- 目录结构
- 注释风格
- 代码组织方式

不要引入个人偏好的不同风格。

---

## 6. 提交规范

### 6.1 每次提交前

说明本次修改的内容和原因。如有影响范围，一并说明。

### 6.2 Commit 粒度

**小步提交**。每个 Commit 聚焦一个独立目标：

- ✅ `完成 URL 解析函数，支持 3 种输入格式`
- ❌ `更新 R1 模块（URL 解析 + GitHub API + Token + 错误处理）`

Commit Message 应让读的人一眼知道改了什么、为什么改。

### 6.3 禁止操作

- `git push --force`
- `git rebase` 改已推送的历史
- 删除重要文件（除非用户明确要求）
- `git add .` 无差别提交（要检查暂存区内容）

## 7. 文档管理

### 7.1 目录结构约定

```
/
├── AGENTS.md              ← 本文档
├── README.md              ← 面向最终用户（后续编写）
├── PRD_OpenCheck.md       ← 产品需求文档
├── WORK_DIVISION.md       ← 分工与 5 日计划
├── docs/
│   ├── CONTRACTS.md       ← 跨角色契约（待创建）
│   └── notes/
│       ├── R1/             ← R1 个人工作文档
│       ├── R2/             ← R2 个人工作文档
│       ├── R3/             ← R3 个人工作文档
│       ├── R4/             ← R4 个人工作文档
│       └── R5/             ← R5 个人工作文档
└── src/                    ← 源代码
```

### 7.2 新增公共文档

**不要随意创建公共文档。** 按以下流程判断：

1. 这个信息是否已有文档覆盖？→ 优先复用，在已有文档中补充
2. 这个信息是否具有**长期维护价值**（不是临时思路或个人记录）？→ 没有长期价值则放在 `/docs/notes/Rx/`
3. 这个信息是否与已有文档功能重复？→ 重复则不创建
4. 确有必要 → 创建新文档，保持 `/docs/` 目录整洁，单一职责（一个主题对应一个文档）

### 7.3 维护公共文档

- 修改代码时，如果影响了公共文档描述的行为，**同步更新文档**，不要让它过期
- 发现文档已过期 → **更新它**，不要忽略
- AI **不为记录自己的思考过程而创建文档**

## 8. 语言约定

| 范围 | 语言 |
|------|------|
| 文档正文、注释 | **中文** |
| 接口字段名 | **英文** |
| 状态枚举值 | **英文**（如 `"pass"` `"partial"` `"fail"`） |
| 代码命名（变量、函数、类型） | **英文** |
| Git Commit Message | **中文**（或中英混合均可，关键是清晰） |

---

## 9. 输出原则

AI 的输出应**优先帮助推进开发**，而不是生成冗长分析。

- ✅ "我把这个函数的错误返回改成了判别联合类型，因为它被 R4 调用时解析不方便"
- ❌ 输出 2000 字分析这个模式的好处、业界现状、其他方案对比（除非用户明确要求）

与当前任务无关的内容，不要输出。

---

## 文档结束

> **本文档是 AI 工作的宪法。** 任何时候不确定该怎么做，回到本文档的 §2.1：选最清晰、最稳定的做法。
