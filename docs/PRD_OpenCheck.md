**关联文档：** [OpenCheck_开源项目体检助手_设计思路.docx](./OpenCheck_开源项目体检助手_设计思路.docx) | [WORK_DIVISION.md](./WORK_DIVISION.md)

## 1. 产品概述

### 1.1 产品定位

OpenCheck 是一个面向开源项目维护者的 Web 自查工具。维护者输入自己的 GitHub 仓库地址后，系统自动读取仓库信息，检测项目的开源规范性——包括关键文件是否齐全、README 文档是否充分、项目结构是否清晰——并生成一份体检报告，给出评分与可操作的改进建议。

**一句话描述：** 给开源项目做一次"体检"，告诉维护者哪里做得不够好、怎么改。

**产品关键词：**
- GitHub 仓库分析
- 开源规范性检测
- 自动化评分
- 改进建议
- Markdown 报告

### 1.2 解决的问题

开源社区中存在大量"裸仓库"——代码扔上去但没有 README、没有 LICENSE、没有运行说明、没有项目结构介绍。这种情况让潜在的贡献者和使用者望而却步，也让项目本身显得不够专业。

OpenCheck 要解决的核心问题是：**让开源维护者用 30 秒就能知道自己的项目在规范性上缺什么、怎么补。**

### 1.3 核心目标

- 用户输入仓库地址后，30 秒内完成检测并展示结果
- 提供清晰的总分 + 逐项得分，让维护者一目了然
- 针对每一项缺失给出可操作的改进建议（而非仅说"缺了"）
- 生成可复制/可下载的 Markdown 报告，方便维护者留存或分享
- 纯浏览器端运行，无需后端服务器，无需部署

### 1.4 技术路线

本项目为纯前端 Web 应用，所有逻辑均在浏览器中运行：

- **数据获取：** 浏览器直接调用 GitHub REST API（`api.github.com`），GitHub 已开放 CORS 支持
- **检测与分析：** 文件存在性判断、README 内容启发式分析、评分计算，均在浏览器中完成
- **数据持久化：** 历史记录与 Token 均存储在浏览器 localStorage 中
- **无需后端服务器：** 没有 API 代理层、没有数据库、没有服务端进程。一个 `npm run dev` 即可启动全部功能

## 2. 产品目标

### 2.1 目标

- 完成完整的检测闭环：输入 → 分析 → 评分 → 建议 → 报告
- 检测覆盖 9 个规范维度（文件存在性 + README 内容完整性）
- 提供总评分（0-100）及等级划分，让结果直观可感
- README 检测具备基础启发式分析能力（而非仅关键词匹配）
- 支持浏览器本地存储检测历史与 Token，刷新不丢失
- 支持用户配置 GitHub Personal Access Token，浏览器直接携带 Token 请求 GitHub API，突破未认证限流

### 2.2 非目标

当前版本明确不做：

- 用户账号系统、注册登录
- 服务端持久化存储（数据库）
- 云端部署、公网访问
- 多仓库批量检测
- 仓库对比功能
- 非 GitHub 平台支持（Gitee、GitLab 等）
- PDF 报告导出（仅支持 Markdown）
- AI 驱动的 README 内容质量分析（作为远期扩展保留可能性）

## 3. 目标用户

### 3.1 核心用户

- **开源项目维护者**（首要用户）：刚发布或正在维护一个 GitHub 开源项目，想自查项目规范性是否达标
- **开源新手**：第一次发布开源项目，不确定开源社区有哪些约定俗成的规范，需要一个清单式指引
- **技术团队负责人**：管理多个内部开源项目，需要快速抽查各仓库的基础规范性

### 3.2 用户诉求

- 快速了解自己仓库的规范性"短板"
- 获得具体、可执行的改进建议（而非抽象评价）
- 检测过程无需复杂配置，输入地址即可
- 结果直观（分数 + 等级），方便在团队内分享
- 希望保留历史检测记录，方便对比改进前后的变化

## 4. 核心使用场景

### 4.1 首次自查

维护者刚听说 OpenCheck，输入自己最近发布的开源项目地址。几秒后看到自己的项目得了 45 分（"需要完善"），系统提示缺少 LICENSE、README 里没有运行说明、目录结构不清晰。维护者根据建议逐一补充，再次检测后提升至 82 分（"较完整"），满意地将报告分享到团队群。

### 4.2 发布前检查

维护者准备将一个内部项目开源，在正式公开前用 OpenCheck 做一次"发布前体检"。检测结果显示 README 缺少部署说明、没有 CONTRIBUTING.md。维护者补齐后确认评分达到 90+，才正式公开仓库。

### 4.3 多项目巡查

技术团队负责人有 5 个开源项目需要维护，逐一输入地址检测，通过浏览器本地历史记录追踪每个项目的评分变化，确保所有项目都维持在"较完整"以上的水平。

### 5.1 仓库地址输入模块

#### 5.1.1 地址输入

用户在一个输入框中输入 GitHub 仓库地址。系统接受以下格式：

- 完整 HTTPS 地址：`https://github.com/owner/repo`
- 省略协议：`github.com/owner/repo`
- 简短格式：`owner/repo`

#### 5.1.2 地址校验

系统对输入进行实时校验，并给出明确的错误提示：

| 校验项 | 错误提示示例 |
|--------|-------------|
| 非空 | "请输入 GitHub 仓库地址" |
| 格式正确（含 owner 和 repo 名） | "地址格式不正确，请输入有效的 GitHub 仓库地址" |
| 仓库存在 | "未找到该仓库，请检查地址是否正确" |
| 仓库为公开仓库 | "暂不支持检测私有仓库" |

校验失败时不发起分析请求，仅在输入框下方显示对应的错误提示。

#### 5.1.3 Token 配置入口

页面提供可选的 GitHub Personal Access Token 配置入口（非强制）。

- Token 由用户自行在 GitHub 设置页生成（提供获取链接），填入后保存在浏览器 localStorage 中
- 检测时，浏览器从 localStorage 读取 Token，直接附带在 `Authorization` 请求头中调用 GitHub API
- 带 Token 的请求限额为 5000 次/小时，不带 Token 为 60 次/小时
- 不填 Token 也可以正常使用，但可能较快触及未认证 API 的调用限制
- 系统仅在请求因速率限制（HTTP 403）被拒绝时，主动提示用户可配置 Token 来提高限额，不会在正常使用时反复催促

### 5.2 仓库基础信息获取模块

系统获取并展示仓库的基本信息，帮助用户在解读检测结果前先了解项目概况。

#### 5.2.1 展示信息

| 信息项 | 说明 |
|--------|------|
| 项目名称 | `owner/repo` 格式 |
| 项目简介 | 仓库的 description 字段 |
| 主要语言 | 仓库占比最高的语言 |
| Star 数 | 仓库关注度 |
| Fork 数 | 仓库被复制的次数 |
| 创建时间 | 仓库首次创建日期 |
| 最近更新时间 | 最近一次提交的时间 |
| 默认分支 | 如 `main` 或 `master` |
| 许可证类型 | 由 GitHub API 识别的许可证（如 MIT、Apache-2.0） |

#### 5.2.2 展示位置

基础信息作为检测结果页的顶部卡片展示，用户一眼可以看到项目的核心元数据。

### 5.3 开源规范检测模块（核心功能）

检测模块是产品的核心，分为两大类检测：**文件存在性检测**和 **README 内容完整性检测**。

#### 5.3.1 文件存在性检测

系统检查仓库根目录下以下文件/目录是否存在：

| 检测项 | 说明 |
|--------|------|
| `README.md` | 项目说明文档（开源项目最基本要求） |
| `LICENSE` | 开源许可证文件 |
| `.gitignore` | Git 忽略规则文件 |
| `CONTRIBUTING.md` | 贡献指南 |
| `CHANGELOG.md` | 更新日志 |
| 依赖声明文件 | 如 `package.json` / `requirements.txt` / `go.mod` / `Cargo.toml` / `pom.xml` 等（根据项目语言判断） |
| `.github/workflows/` | CI/CD 工作流配置目录（非空目录才算存在） |

每项检测结果为"通过"或"未通过"二值。

#### 5.3.2 README 内容完整性检测

系统读取 `README.md` 文件全文，进行启发式分析，判断以下内容是否被覆盖：

| 检测维度 | 判定逻辑（启发式） |
|----------|-------------------|
| **运行说明** | README 是否包含安装依赖、启动/运行项目的步骤描述。对代码块数量、关键段落长度、关键词（"安装"/"启动"/"运行"/"build"/"install"/"run"）进行加权判断 |
| **技术栈说明** | README 是否明确列出项目使用的技术/语言/框架。通过检测技术名词密度、是否有"技术栈"/"Tech Stack"/"Built With"等章节标题综合判断 |
| **项目结构说明** | README 是否描述了项目的目录结构或模块划分。通过检测目录树格式文本（如 `├──`、`src/`、`├` 符号）、"项目结构"/"目录结构"/"Project Structure"等章节标题，以及目录路径列表密度来综合判断 |
| **部署说明** | README 是否包含部署相关的步骤或配置说明。通过检测关键词（"部署"/"deploy"/"发布"/"release"）及相关段落的长度综合判断 |
| **截图/演示** | README 中是否包含图片（检测 `![...](...)` 图片链接语法或 `<img>` 标签）。包含至少一张图片即视为通过 |
| **使用说明** | README 是否包含 API 文档、配置说明、示例代码等内容，不只是简单介绍。通过整体内容丰富度（总长度、代码块数量、章节数量）来综合判断 |

每项检测结果为"通过"、"部分通过"或"未通过"三值，而非简单二值判断。"部分通过"表示有相关内容但不充分（如只有一句话提到运行方式但没有具体步骤）。

#### 5.3.3 检测结果汇总

所有检测项的结果汇总在一个列表中展示，每一项显示：
- 检测项名称
- 检测结果（通过/部分通过/未通过）
- 分值（得了多少分 / 满分多少分）

### 5.4 评分模块

#### 5.4.1 评分规则

每个检测项有预设分值，所有分值的总和为 100 分。MVP 阶段固定分值分配如下：

| 检测项 | 满分 | 类别 |
|--------|------|------|
| README.md 存在 | 20 | 文件存在性 |
| LICENSE 存在 | 15 | 文件存在性 |
| 运行说明 | 15 | README 内容 |
| 技术栈说明 | 10 | README 内容 |
| 项目结构说明 | 10 | README 内容 |
| .gitignore 存在 | 10 | 文件存在性 |
| CONTRIBUTING.md 存在 | 10 | 文件存在性 |
| 部署说明 | 5 | README 内容 |
| CHANGELOG.md 存在 | 5 | 文件存在性 |

- 文件存在性检测：通过得满分，未通过得 0 分
- README 内容检测：通过得满分，部分通过得一半分（向下取整），未通过得 0 分

总分 = 所有检测项得分之和。

#### 5.4.2 评分等级

| 分数区间 | 等级 | 含义 |
|----------|------|------|
| 90 – 100 | 优秀 | 项目规范性很高，适合作为同类项目的参考标杆 |
| 75 – 89 | 较完整 | 主要规范项已覆盖，个别细节可进一步完善 |
| 60 – 74 | 基本可用 | 核心文件齐全，但文档丰富度不足，建议补充 |
| 60 以下 | 需要完善 | 存在明显缺失，建议按改进建议逐项补充 |

### 5.5 改进建议模块

#### 5.5.1 自动生成建议

系统根据检测出的缺失项/不完整项，自动匹配预设的建议文案。MVP 阶段使用固定模板，不接入 AI。

每条建议包含：
- **对应的缺失项**：让用户知道这条建议针对哪个问题
- **建议内容**：用一两句话说明应该做什么、为什么重要

#### 5.5.2 建议模板示例

| 缺失项 | 建议内容 |
|--------|---------|
| 缺少 LICENSE | 建议添加开源许可证（如 MIT、Apache-2.0、GPL-3.0），明确他人使用、修改和分发代码的权限。GitHub 创建新仓库时可一键添加。 |
| 缺少运行说明 | 建议在 README 中补充"快速开始"章节，包括：安装依赖的命令、启动项目的命令、访问地址（如有 Web 界面）。 |
| 缺少项目结构说明 | 建议在 README 中说明主要目录的用途（如 `src/`、`components/`、`api/`、`utils/`），帮助新贡献者快速找到对应代码。 |
| 缺少 CONTRIBUTING.md | 建议添加贡献指南，说明如何提交 Issue、如何发起 Pull Request、代码风格要求等，降低外部贡献者的参与门槛。 |
| 缺少 .gitignore | 建议添加 .gitignore 文件，避免将编译产物、依赖目录、IDE 配置文件等提交到仓库。GitHub 提供了常用语言的模板。 |
| 缺少 CHANGELOG.md | 建议添加更新日志，按版本记录新增功能、修复的 Bug、Breaking Changes，方便用户了解项目演进。 |
| 缺少截图/演示 | 建议在 README 中添加项目截图或演示 GIF，让潜在用户在阅读文字前先直观了解项目的样子。 |
| 缺少部署说明 | 建议补充部署章节，说明如何将项目部署到生产环境（如 Docker 部署、Vercel/Netlify 一键部署等）。 |

#### 5.5.3 建议展示

改进建议模块展示在检测结果页面下半部分，仅展示得分未满的检测项对应的建议（满分项不展示建议）。如果所有项都是满分，显示"你的项目在检测范围内没有明显缺失，继续保持！"。

### 5.6 报告生成模块

#### 5.6.1 报告格式

系统将检测结果自动编排为 Markdown 格式的文本。报告包含以下章节：

1. **报告头部**：项目名称、检测时间
2. **项目基本信息**：仓库基础信息摘要
3. **检测总评分**：分数 + 等级
4. **检测结果明细**：每一项的名称、结果、得分
5. **改进建议**：列出所有需要改进的项及对应建议
6. **报告尾部**：Powered by OpenCheck 标识

#### 5.6.2 报告操作

在报告页面提供两个操作：

- **复制报告**：一键将 Markdown 全文复制到系统剪贴板，操作成功后给出"已复制"提示
- **下载报告**：将 Markdown 内容保存为 `.md` 文件并触发浏览器下载，文件名格式为 `opencheck-{owner}-{repo}-{日期}.md`

### 5.7 历史记录模块

#### 5.7.1 存储方案

- 检测完成后，自动将本次检测结果保存到浏览器的 localStorage 中
- 数据以仓库地址为唯一标识，同一仓库重复检测时覆盖旧记录
- 存储数据量小（每次检测仅保存关键字段），不设存储上限（由浏览器自身限制）

#### 5.7.2 保存内容

每条历史记录包含：

- 仓库地址
- 仓库名称（`owner/repo`）
- 检测总评分
- 检测等级
- 检测时间戳
- 检测结果摘要（仅保存各检测项的通过/未通过状态，不保存完整报告正文）

#### 5.7.3 历史记录列表

提供独立的历史记录页面，展示已检测过的仓库列表。每条记录显示：
- 仓库名称
- 最近一次评分 + 等级标签
- 最近一次检测时间

列表按检测时间倒序排列（最近检测的排在前面）。

#### 5.7.4 历史记录操作

| 操作 | 行为 |
|------|------|
| 点击某条记录 | 跳转到该仓库的检测结果页（展示最后一次检测的结果） |
| 重新检测 | 在历史记录详情页提供"重新检测"按钮，点击后重新拉取仓库信息并覆盖旧记录 |
| 删除单条记录 | 每条记录提供删除按钮，点击后从 localStorage 中移除该条 |
| 清空全部记录 | 列表页提供"清空历史"按钮，点击后弹出确认对话框，确认后清除所有记录 |

#### 5.7.5 边界情况

- 用户首次使用时历史记录为空，展示空状态提示："还没有检测记录，输入仓库地址开始第一次检测吧"
- 如果 localStorage 被清理（浏览器清除数据），历史记录自然丢失，不影响核心检测功能

## 6. 页面设计

### 6.1 页面清单

| 页面 | 主要内容 |
|------|---------|
| **首页** | 产品标题与简介、仓库地址输入框、"开始检测"按钮、简单的功能介绍文案。整体风格干净、聚焦于输入操作 |
| **检测结果页** | 仓库基本信息卡片、总评分环/大数字 + 等级标签、检测项列表（各项名称、状态标识、得分）、改进建议列表。页面较长需合理分区 |
| **报告页** | 自动生成的 Markdown 预览、复制按钮、下载按钮 |
| **历史记录页** | 历史检测仓库列表、每条记录的基本信息、清空历史按钮。入口放在首页或顶部导航 |
| **Token 配置页/弹窗** | 简洁的 Token 输入界面，说明 Token 用途和获取方式 |

### 6.2 交互原则

- **反馈明确**：检测过程中展示加载状态（进度条或骨架屏），不要让用户干等。检测完成后平滑过渡到结果页
- **错误友好**：网络错误或仓库不存在时，给出清晰的错误说明和重试入口，不要让用户面对空白页
- **操作可逆**：历史记录删除前需确认，避免误操作
- **移动端可用**：页面布局需在手机浏览器上正常显示（即使主要使用场景是桌面端，不排除移动端打开的情况）

### 6.3 视觉风格建议

- 整体风格：简洁专业，偏工具型产品调性
- 配色以中性色为主，用少量强调色标识评分等级（如绿色=优秀，蓝色=较完整，橙色=基本可用，红色=需要完善）
- 检测项的状态用清晰的图标区分（如 ✓ 通过、△ 部分通过、✗ 未通过）
- 不追求花哨动效，以信息清晰传达为第一优先级

## 7. 数据设计

### 7.1 检测输入

| 字段 | 类型 | 说明 |
|------|------|------|
| 仓库地址 | 文本 | 用户输入的原始地址 |
| 解析后的 owner | 文本 | 从地址中解析的用户名/组织名 |
| 解析后的 repo | 文本 | 从地址中解析的仓库名 |
| Token（可选） | 文本 | 用户配置的 GitHub Personal Access Token |

### 7.2 检测结果

单次检测的完整结果：

| 字段 | 类型 | 说明 |
|------|------|------|
| 检测时间 | 时间戳 | 检测完成的时间 |
| 仓库基本信息 | 对象 | 包含 5.2.1 中列出的各项信息 |
| 检测项列表 | 数组 | 每项包含：名称、类别（文件/内容）、结果（通过/部分通过/未通过）、得分、满分 |
| 总评分 | 整数 | 0-100 |
| 评分等级 | 文本 | 优秀/较完整/基本可用/需要完善 |
| 改进建议列表 | 数组 | 每项包含：对应的缺失项名称、建议内容 |

### 7.3 历史记录（localStorage）

每条历史记录：

| 字段 | 类型 | 说明 |
|------|------|------|
| 仓库地址 | 文本 | 用于去重的唯一标识 |
| 仓库名称 | 文本 | 用于列表展示 |
| 总评分 | 整数 | 最近一次检测的分数 |
| 评分等级 | 文本 | 最近一次检测的等级 |
| 检测时间 | 时间戳 | 最近一次检测的时间 |
| 检测结果摘要 | 对象 | 各检测项的通过/未通过状态（不保存完整报告） |

## 9. 明确不做

以下功能在当前及后续版本中均明确排除：

- 用户账号系统、注册登录
- 云端同步、多设备历史记录
- 私有仓库检测（仅支持公开仓库）
- 付费功能或广告
- 批量检测多个仓库
- 自动修复（如自动生成 README 并提交 PR）
- 任何形式的社交功能（评论、分享社区等）
